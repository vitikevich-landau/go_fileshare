## Подробные замечания

### CR-01 — точки входа исключены из Git

Приоритет: **P1**  
Файлы: `.gitignore:5-6`, `Dockerfile:8`

Шаблоны:

```gitignore
fshare-daemon
fshare-commander
```

совпадают не только с бинарниками, но и с одноимёнными компонентами пути.
Из-за этого `git status --ignored cmd` показывает `!! cmd/`, а `git ls-files
cmd` не возвращает файлов.

Последствия:

- чистый clone не содержит исходников обоих приложений;
- команда из README `go build ./cmd/fshare-daemon ./cmd/fshare-commander`
  завершается ошибкой;
- Dockerfile не собирается из чистого clone;
- CI может проверять только библиотеки и не замечать отсутствие приложений.

Направление исправления:

- привязать правила к корню или конкретному бинарному файлу, например
  `/fshare-daemon`, `/fshare-commander` либо использовать только `/bin/`;
- добавить оба `cmd/.../main.go` в Git;
- добавить CI-проверку явной сборки обоих приложений и Docker image.

Критерий закрытия:

1. `git ls-files cmd` показывает обе точки входа.
2. Сборка из нового clone проходит.
3. `docker build .` проходит без локальных игнорируемых файлов.

### CR-02 — неполная загрузка может завершиться успехом

Приоритет: **P1**  
Файлы: `internal/client/client.go:360-375`, `internal/server/download.go:113-128`

При получении `DOWNLOAD_DONE` клиент:

- не проверяет `received == total`;
- не проверяет `DownloadDone.TransferID == DownloadAccept.TransferID`;
- проверяет checksum только когда `Algo == SHA256`;
- для `AlgoPending` и неизвестного алгоритма сразу переименовывает `.part`.

Сервер при этом может выйти из цикла по раннему EOF до `total_size`, а при
ошибке вычисления checksum специально отправляет `AlgoPending` и нулевой hash.

Последствия: усечённый или вообще непроверенный файл может быть опубликован под
итоговым именем, а CLI/TUI сообщит об успешной загрузке.

Направление исправления:

- клиенту проверять transfer ID и точное число байт;
- отклонять `DOWNLOAD_DONE` без поддерживаемого обязательного checksum;
- серверу не отправлять успешный `DOWNLOAD_DONE` после short read или ошибки
  checksum; возвращать протокольную ошибку;
- желательно вычислять checksum по тому же открытому файловому объекту или
  фиксированному snapshot, который использовался для передачи.

Тесты для закрытия:

- server сообщает `total=100`, отправляет 50 байт и `DOWNLOAD_DONE`;
- `DOWNLOAD_DONE` с другим transfer ID;
- `DOWNLOAD_DONE` с `AlgoPending`;
- файл обрезается или заменяется во время передачи.

Во всех случаях итоговый файл не должен появляться, а `.part` должен оставаться
либо удаляться по явно выбранной политике.

### CR-03 — idle timeout обрывает активную загрузку

Приоритет: **P1**  
Файлы: `internal/server/conn.go:145-151`, `internal/tui/model.go:597-616`

После `DOWNLOAD_REQUEST` передача идёт в отдельной goroutine, а request loop
сразу ставит read deadline `IdleTimeoutS` и ждёт следующее входящее сообщение.
Исходящие `CHUNK_DATA` deadline не обновляют.

TUI на всё время `Client.Download` удерживает `clientMu`, поэтому event pump не
может вызвать `Ping`. Batch-клиент heartbeat во время загрузки также не посылает.

Последствия: любая активная загрузка, продолжающаяся дольше idle timeout
(по умолчанию 600 секунд), будет закрыта сервером как простаивающая.

Направление исправления:

- не применять idle timeout к сессии с активной передачей; либо
- учитывать исходящую активность и обновлять deadline; либо
- обеспечить независимый heartbeat, не конкурирующий за client read ownership.

Критерий закрытия: integration test с коротким idle timeout и искусственно
замедленной передачей должен завершать файл успешно после нескольких timeout
интервалов.

### CR-04 — неограниченный handshake и PBKDF2 cost от сервера

Приоритет: **P1**  
Файлы: `internal/client/client.go:71-123`, `internal/auth/scram.go:27-37`

`DialTimeout` ограничивает только установление TCP-соединения. После connect
клиент не устанавливает deadline на чтение `HELLO_OK` и `AUTH_OK`.

Кроме того, поле `PBKDF2Iters` из недоверенного `HELLO_OK` преобразуется в `int`
и без верхней границы передаётся в PBKDF2. Максимальное wire-значение —
4 294 967 295 итераций.

Последствия: сервер или MITM может бесконечно не отвечать либо заставить клиент
надолго загрузить одно ядро CPU. TLS в текущей версии отсутствует, поэтому
сценарий MITM входит в заявленную модель угроз.

Направление исправления:

- устанавливать deadline на весь handshake, затем очищать его;
- валидировать auth mode и разумный диапазон PBKDF2 iterations;
- возвращать обычную ошибку вместо panic/неограниченного вычисления.

Тесты для закрытия: молчащий peer, частичный `HELLO_OK`, нулевой iteration count,
значение выше установленного максимума.

### CR-05 — memory DoS до аутентификации

Приоритет: **P1**  
Файлы: `internal/proto/frame.go:22-36`, `internal/config/settings.go:64-68`

`ReadFrame` принимает любой известный message type и сразу выделяет payload до
`MaxControlPayload` — 4 MiB. Проверка размера конкретного сообщения выполняется
только позднее при decode.

При стандартных 200 соединениях атакующий без учётной записи может отправить
только пятибайтные заголовки `HELLO` с длиной 4 MiB и удерживать около 800 MiB
памяти до handshake timeout, непрерывно заменяя закрытые соединения.

Направление исправления:

- в handshake читать заголовок с отдельным малым лимитом;
- ввести per-message maximum payload до allocation;
- при необходимости ограничить суммарный объём входящих незавершённых frames.

Критерий закрытия: нагрузочный тест с большим числом заголовков и без payload не
должен приводить к пропорциональному выделению 4 MiB на соединение.

### CR-06 — гонка при проверке лимита сессий

Приоритет: **P2**  
Файл: `internal/server/conn.go:133-140`

`countUser` и `setAuthed` выполняются раздельно. Несколько параллельных валидных
handshake одного пользователя могут одновременно увидеть значение ниже лимита,
после чего все станут authenticated.

Направление исправления: объединить проверку и резервирование user session slot
в одной атомарной операции registry/session manager; освобождать reservation при
ошибке или закрытии соединения.

Критерий закрытия: конкурентный test запускает много одновременных входов при
лимите 1; успешным остаётся ровно один.

### CR-07 — утечка admin config events обычному пользователю

Приоритет: **P2**  
Файлы: `internal/server/conn.go:180-182`, `internal/server/session.go:243-249`

Сообщение `SUBSCRIBE` разрешено роли user. Переданная маска сохраняется целиком,
а broadcast проверяет только совпадение бита. Поэтому обычный пользователь может
установить `SubConfig` и получать `EVENT_CONFIG`, хотя спецификация ограничивает
этот поток администраторами.

Направление исправления: фильтровать маску по роли в `dispatch` либо повторно
проверять роль при broadcast чувствительного типа событий.

Критерий закрытия: user с маской `SubConfig` не получает событие после admin set;
admin с той же маской получает.

### CR-08 — hot `log.level` не применяется

Приоритет: **P2**  
Файлы: `internal/config/hub.go:152-155`, `cmd/fshare-daemon/main.go:82-84`

`log.level` объявлен hot, сохраняется в snapshot и возвращает администратору
результат `applied`. Реальный `slog.Logger` создаётся один раз со статическим
`HandlerOptions.Level`; последующих чтений `cfg.Log.Level` нет.

Направление исправления: создать logger с `slog.LevelVar` и обновлять её через
callback Hub либо перевести настройку в restart-only.

Критерий закрытия: после ADMIN_SET с `debug` сообщения debug начинают попадать в
тестовый handler без restart; после возврата к `info` снова фильтруются.

### CR-09 — checksum cache допускает устаревший hash

Приоритет: **P2**  
Файл: `internal/vfs/vfs.go:287-305`

Cache hit определяется тройкой `(path, size, mtime)`, где mtime округлён до целых
Unix-секунд. Файл, изменённый с тем же размером в пределах одной секунды или с
сохранённым timestamp, получит старый SHA-256. Watcher снижает вероятность, но не
гарантирует корректность при restart, отключённых событиях или пропущенном event.

Направление исправления:

- хранить как минимум nanosecond mtime;
- при доступности учитывать file identity/change time;
- определить поведение при невозможности надёжно подтвердить cache entry.

Критерий закрытия: два разных содержимых одинакового размера и с одинаковым
секундным mtime должны давать разные checksum, включая reload cache с диска.

### CR-10 — ответ большого каталога превышает лимит протокола

Приоритет: **P2**  
Файлы: `internal/server/conn.go:201-207`, `internal/proto/frame.go:28-34`

VFS возвращает весь каталог, затем сервер кодирует его в один
`LIST_DIR_RESPONSE`. Исходящий размер не проверяется. Если payload превышает
4 MiB, клиент отвергает заголовок как oversize и соединение становится
непригодным.

Направление исправления: добавить пагинацию/cursor в протокол либо перед encode
ограничивать число записей так, чтобы ответ гарантированно помещался в frame, и
возвращать явную ошибку вместо отправки недопустимого кадра.

Критерий закрытия: каталог, чей полный listing больше 4 MiB, можно просматривать
страницами или получить контролируемую protocol error без разрыва соединения.

## Задокументированные ограничения, не включённые в defects

- TLS пока не реализован, поэтому содержимое файлов и admin traffic передаются
  открытым текстом.
- При отсутствии `users.json` включается bootstrap mode: любой login получает
  роль admin.

До появления TLS и безопасного bootstrap сервер не следует публиковать в
недоверенную сеть без внешнего защищённого туннеля и сетевых ACL.

## Повторное ревью (round 3): R3-1…R3-7 — исправлены

Третий проход по head `076196c` показал, что исправления RR-1…RR-6 присутствуют,
но 7 пограничных сценариев остались незакрытыми. Все исправлены (по коммиту на
пункт, с регрессионными тестами):

| ID | Приоритет | Замечание | Исправление |
|---|---|---|---|
| R3-1 | P1 | Cancel зависал внутри rate limiter: `limiter.Wait` не будился на client-cancel, при низком `per_client_bps` ожидание одного chunk длилось минуты | Rate-limit watcher теперь селектит и на `<-cancel` (будит `Wait` через `ctxCancel`); после возврата `Wait` поток отличает client-cancel (шлёт терминальный `ERROR(CANCELLED)`) от teardown |
| R3-2 | P1 | Сервер игнорировал `TransferID` при cancel; гонка late-cancel могла оборвать следующую передачу | `cancelDownload(tid)` сверяет `tid` с активным `cancelTID`; клиентский `DownloadCtx` дожидается cancel-watcher (`WaitGroup`) перед возвратом, чтобы поздний `DOWNLOAD_CANCEL` не утёк в следующую передачу |
| R3-3 | P2 | Локальная ошибка после `ACCEPT` (open/write `.part`) оставляла соединение рассинхронизированным | Любой локальный/аномальный ранний выход после `ACCEPT` вызывает `abortConn` (закрывает соединение); терминальные кадры сервера (`DONE`/`ERROR`, включая graceful RR-3 cancel) соединение не рвут |
| R3-4 | P2 | SIGHUP и `ADMIN_SET log.level` могли рассинхронизировать snapshot и живой logger (без data race) | `Hub.ApplyWith` применяет runtime-эффект (LevelVar) под тем же writer lock, что и swap snapshot — линеаризовано с `Hub.Set` |
| R3-5 | P2 | На non-Linux RR-5 не работал: возврат `0` сводил ключ к `(path,size,mtime)`, тест делал `Skip` | `changeTimeNanos` возвращает `(nanos, reliable)`; добавлен darwin (`Ctimespec`); где change-time ненадёжен (Windows) — кэш не отдаёт hit, а пересчитывает; тест больше не скипается |
| R3-6 | P2 | Watchdog считал долгий синхронный `Checksum`/`List`/admin-обработчик простоем и мог оборвать соединение посреди запроса | `serveRequests` выставляет `inFlight` вокруг `dispatch` и `touch()` после; `reapable` не жнёт при `downloading` или `inFlight` |
| R3-7 | P1 | `PollEvents` терял частично прочитанный frame: deadline стоял на весь `ReadFrame` | Deadline применяется только до первого байта (`conn.Read(first[:1])`); дальше `proto.ReadFrameContinue` дочитывает кадр без deadline — частичный frame не теряется |

Полный прогон `go build ./...`, `go vet ./...`, `go test -race ./...` зелёный;
кросс-сборка `GOOS=linux` и `GOOS=darwin` проходит.
