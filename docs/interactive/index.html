Что такое FShare Commander — за 90 секунд

go_fileshare — клиент-серверная файлораздача на Go со своим бинарным протоколом. Два бинаря: демон, который раздаёт дерево каталогов по TCP, и полноэкранный TUI-клиент в стиле Midnight Commander. Это переписанный «с нуля» на Go порт C++-проекта fileshare v2; провод воспроизведён байт-в-байт, поэтому Go-сервер обслуживает эталонного C++-клиента и наоборот.

fshare-daemon — сервер
Неинтерактивный демон: раздаёт дерево share-root через VFS, аутентификация challenge–response, push-события об изменениях файлов, живой конфиг (лимиты скорости) без перезапуска, корректная остановка, перезагрузка по SIGHUP.
fshare-commander — клиент
Две панели, горячие клавиши, докачка с прогрессом/скоростью/ETA, подсветка нового, индикатор связи + авто-реконнект, админ-панель по F9. Тот же бинарь работает и как --batch для скриптов.

Три идеи, на которых всё держится

1. Session — центральная сущность. До успешного AUTH соединение в состоянии HANDSHAKE и принимает только HELLO/AUTH_*/PING; таймаут рукопожатия (10 с) выкашивает полуоткрытые «слоу-лорис». После входа диспетчер — единственная точка авторизации: у каждого типа сообщения объявлена минимальная роль (anonymous/user/admin), и захваченная, но не-админская сессия отвергается сервером независимо от того, что нарисовал UI.
2. Горячий конфиг через снапшоты. Hub хранит указатель на неизменяемый снимок настроек в atomic.Pointer. Горячий путь (каждый чанк, каждый accept) делает одну атомарную загрузку без блокировок. Писатель строит новый снимок, валидирует и атомарно подменяет — гонок нет по построению, и смена лимита скорости тормозит уже идущую закачку без разрыва.
3. Сеть строго вне UI-потока. В клиенте вся сеть живёт в фоновых горутинах, а с UI общается через один канал событий; в сервере единственный writeLoop на сессию делает кадры на проводе атомарными, а рассылки шлются неблокирующе и отбрасываются, если очередь клиента переполнена, чтобы один тормоз не задержал остальных.

Разделы 2–4 — про то, из чего система собрана (пакеты, байты кадра, карта сообщений). Разделы 5–6 — как она работает (вход и сквозные сценарии). Разделы 7–10 — почему так (структуры, конкурентность, безопасность, план). Прыгать по разделам можно клавишами 1…0.

Карта пакетов

Клик по пакету — его роль, поля и связи. Стрелки внизу = «кто кого импортирует» (по реальным import'ам).

Выбери пакет, чтобы увидеть его роль и зависимости.

Слои зависимостей

основа (только stdlib):     proto ·   config ·   ratelimit
домен над протоколом:       vfs → proto ·  auth → proto ·  watcher → proto
транспорт:                  client → auth, proto

В каждом пакете есть файл types.go — «словарь предметной области»: доменные величины сделаны псевдонимами базовых типов (type SessionID = uint64), поэтому сигнатуры читаются как утверждения о протоколе, а переименование поля не меняет ни одного байта на проводе.

Кадр по байтам

Всё общение идёт кадрами: 5-байтовый заголовок (тип + длина тела) и тело. Все числа — big-endian. Выбери сообщение и наведи (или тапни) на байт — каждый знает, что он кодирует.

┌────────────┬───────────────────────┬──────────────────────┐
│ msg_type   │ payload_length        │ payload              │
│ 1 байт u8  │ 4 байта u32, big-endian│ payload_length байт  │
└────────────┴───────────────────────┴──────────────────────┘
заголовок число длина строки строка (UTF-8) перечисление фикс. байты сырые данные
Наведи на байт.
Каждый байт знает свою роль: заголовок, число big-endian, префикс длины строки, сама строка, фиксированное поле (challenge/proof/checksum) или сырой хвост.

Правила и пределы (как в коде)

  • Строка = длина u16 + UTF-8 байты. Два исключения: ADMIN_CONFIG использует u32-префикс (JSON конфига), а CHUNK_DATA несёт сырые байты без префикса — их длину задаёт длина кадра (payload_length − 4).
  • Строгий разбор: после тела курсор обязан быть ровно в конце (require_end) — лишний байт означает рассинхрон и рвёт только это соединение. Каждый доступ к буферу проверяет границы и возвращает ошибку вместо паники: битый кадр не роняет процесс.
  • Потолки до аллокации: неизвестный тип отвергается ещё до чтения тела; счётчик записей проверяется против MaxListEntries=1<<20 перед выделением; тело большого сообщения (листинг/конфиг) ограничено 4 МиБ, чанк — 64 КиБ, а дорукопожатные кадры — жёстким малым лимитом (~64 КиБ), чтобы аноним не заставил сервер что-то выделять.

Карта сообщений

Семейство сообщения читается по старшему полубайту кода. Выбери группу — коды, направление, тело и минимальная роль. Это единый словарь запросов и ответов.

Состояния соединения

CONNECTED ──HELLO──▶ HELLO_RECEIVED ──AUTH_REQUEST──▶ AUTHED(role)
    │                     │                              │
 только HELLO/PING     только AUTH/PING             всё по роли (MinRole)
    │  всё под одним дедлайном рукопожатия (10 с)        │  дедлайн снят: блокирующее
    ▼  чужой первый байт → ERROR(UNSUPPORTED_VERSION)    ▼  чтение без пофреймового таймаута (RR-1)

Диспетчер сверяет роль сессии с MinRole(тип): HELLO/AUTH_REQUEST/PING/PONG/ERROR — anonymous; LIST/STAT/CHECKSUM/DOWNLOAD_REQUEST/DOWNLOAD_CANCEL/SUBSCRIBE — user; всё остальное (включая любой ответ/событие сервера и весь админ-канал) по умолчанию требует admin, а серверные коды от клиента вдобавок отвергаются как BAD_REQUEST. Роль ниже нужной → ERROR(ACCESS_DENIED), но соединение живёт.

Вход: SCRAM challenge–response

Два свойства объясняют всю конструкцию: пароль никогда не пересекает сеть (ни текстом, ни хешем), и кража users.json не даёт войти — в базе лежит лишь необратимый верификатор StoredKey. Выбери шаг — подсветятся значения, которые считает клиент, и то единственное, что хранит сервер.

Почему кража базы бесполезна

В users.json хранится только StoredKey = SHA256(ClientKey). Чтобы подделать ClientProof, нужен ClientKey — а он выводится из пароля и в базе его нет. Из StoredKey нельзя восстановить ни пароль, ни ClientKey. Одноразовый challenge (16 байт, крипто-random) делает и перехваченное доказательство бесполезным: его нельзя переиграть на новый вызов.

Детали, важные на практике. Растяжение пароля — PBKDF2-HMAC-SHA256 (порог 600 000 итераций; ниже — конфиг предупреждает при старте). Финальное сравнение в Verify — константного времени (subtle.ConstantTimeCompare), чтобы по длительности нельзя было подбирать байты. Число итераций сервер объявляет в HELLO_OK; клиент зажимает его в [1, 10 000 000], чтобы враждебный сервер не сжёг ему CPU (CR-04). Гард банит IP после N неудач + пауза 1 с на каждый отказ; входы и баны — в аудит.

Порт сознательно взял PBKDF2 вместо Argon2id из ТЗ — чтобы стандартная сборка аутентифицировала «из коробки», без libsodium. TLS шифрования трафика на этом этапе ещё нет (см. раздел «безопасность»): v2.0 защищает учётные данные, а не содержимое.

Структуры данных

Не таблицы полей ради таблиц, а роль каждой структуры, её ключевые поля и дисциплина доступа (что чем защищено). Полные словари — в types.go каждого пакета.

server.Session — один подключённый клиент

Центральная сущность демона. Исходящие кадры идут через канал out, который вычитывает единственная горутина-писатель — поэтому кадры атомарны, а медленный клиент тормозит только свои передачи.

Под мьютексом mu (редкие изменения): login, role, authed, cancelDL/cancelTID (канал отмены и id его передачи — R3-2).
Атомарные (горячий путь, без блокировок): subMask (подписка), bytes (отдано), curPath (где клиент), downloading, inFlight (идёт ли обработчик — R3-6), lastActivity (для сторожа простоя).
Каналы: out (глубина 64, backpressure), done (закрывается при teardown — сигнал всем горутинам сессии).

send блокируется для backpressure; trySend — неблокирующая (рассылки). Отмена срабатывает только если TransferID совпал с активной передачей — «поздний» DOWNLOAD_CANCEL не убьёт следующую закачку.

server.Registry — реестр живых сессий

Карта SessionID → *Session под своим мьютексом. К ней обращаются из разных горутин: диспетчер, рассылки, админ-kick, сторож. Ключевой метод reserveUserSlot атомарно проверяет лимит сессий на пользователя и помечает сессию аутентифицированной — так параллельные рукопожатия одного логина не проскочат проверку все разом (CR-06). broadcast шлёт кадр каждой подписанной сессии неблокирующе (drop-on-full).

config.Hub + Settings — снапшоты горячего конфига

Settings — плоская value-структура (без указателей/срезов/карт), поэтому её снимок — дешёвая копия по значению. Hub держит atomic.Pointer[Settings]: Current() — одна атомарная загрузка, никогда не блокирует. Писатели сериализованы мьютексом wmu: копируют снимок, применяют ключ, валидируют, атомарно подменяют указатель и под тем же локом зовут onChange (сохранить на диск + разослать EVENT_CONFIG) — чтобы файл и рассылка отражали один снимок.

Ключи делятся на hot (лимиты, тайм-ауты, motd, уровень лога) и restart (порт, share_root, воркеры, users_file, pbkdf2_iters, параметры watcher). Попытку сменить restart-ключ на лету Set отвергает понятной ошибкой, а не молча принимает бесполезное изменение.

vfs.VFS — дерево раздачи + ленивый кэш сумм

Раздаёт дерево, запертое внутри одного share-root через os.Root (Go 1.24+): даже «..» или симлинк наружу не выведут операцию за корень — класс уязвимостей TOCTOU закрыт по построению. Путь живёт в трёх формах: VirtualPath (как прислал клиент) → CleanedPath (нормализован, уходит в протокол) → OS-relative (для os.Root).

Два независимых мьютекса: mu стережёт cache (checksum-кэш по ключу path,size,mtime-нс,ctime) и флаг dirty; statsMu — фоновый снимок статистики раздачи. Хеширование идёт вне лока: два параллельных промаха могут оба пересчитать — это безобидно (побеждает последний). Сумма считается лениво и с учётом отмены контекста (R4-3).

ratelimit.Limiter — двухуровневый token bucket

Ограничение скорости на двух уровнях сразу: одно глобальное ведро на весь сервер и по ведру на логин (не на сокет!) — поэтому N параллельных закачек одного пользователя делят один бюджет (§8 bug 11). Лимиты читаются на каждом чанке, поэтому смена конфига перенастраивает уже идущую передачу через SetLimit без разрыва.

Единственный мьютекс держится только на короткие секции (создать/перенастроить/пометить активным); блокирующий WaitN идёт вне лока. Счётчик active не даёт сборщику удалить ведро, пока в нём кто-то ждёт; фоновый reaper выселяет вёдра, простаивавшие >10 мин — чтобы карта не росла без границ.

watcher.Watcher — файловые события

Превращает изменения под корнем в EVENT_FS, оборачивая fsnotify и добавляя две вещи: рекурсию (fsnotify следит за одной папкой — watcher подписывается на каждую поддиректорию и на вновь созданные, снимая подписки при удалении, чтобы их набор не рос — §8 bug 15) и debounce (одно сохранение файла = пачка событий; watcher откладывает выдачу и шлёт одно итоговое). Наружу уходит только VirtualPath; путь вне корня события не рождает.

tui.Model — единственный источник правды UI

Архитектура Bubble Tea (паттерн Elm): всё состояние UI в одной Model, меняется только в Update(msg), рисуется только в View (чистая функция). Глобальных переменных нет — логика панелей тестируется без терминала. Panel — чистые переходы состояния (курсор, прокрутка, выделение по имени, сортировка).

Один мьютекс clientMu сериализует весь ввод-вывод к блокирующему транспорту (насос, закачка, листинг, checksum, админ). Один канал events (буфер 64) — единственный мост из фоновых горутин в Update. Хитрость: Close() клиента зовётся вне лока (закачка держит clientMu весь трансфер), чтобы разблокировать зависшее чтение — и лишь потом client=nil под локом.

client.Client — блокирующий транспорт

Им пользуются и TUI, и --batch. Проводит рукопожатие, шлёт запросы и качает файлы, прозрачно переправляя асинхронные EVENT_*/PONG в обработчик событий, пока ждёт конкретный ответ — даже посреди закачки, не портя поток. Запросные методы нельзя звать конкурентно (по очереди пишут/читают один сокет), а Ping/Interrupt/Close — можно из другой горутины (heartbeat и отмена).

Модель конкурентности

Одна картинка на сервере: несколько горутин на соединение с чётким разделением ролей, чтобы медленный клиент тормозил только себя.

handleConn (владелец соединения)
  ├─ writeLoop      ЕДИНСТВЕННЫЙ писатель в сокет ← канал out (кадры атомарны)
  ├─ serveRequests  ЕДИНСТВЕННЫЙ читатель: разбор кадра → проверка роли → обработчик
  ├─ idleWatchdog   закрывает молчащее соединение (но не во время закачки/обработки)
  └─ streamFile     (на время закачки) отдаёт файл чанками — читатель свободен для PING/CANCEL

Правила, действующие по всей системе

  • Горячий конфиг — снапшоты. Два источника сходятся в один писатель под общим локом. ADMIN_SET по сети: валидировать → атомарно подменить → сохранить на диск → разослать EVENT_CONFIG подписанным админам. SIGHUP: перечитать файл → валидировать → атомарно подменить, сохранив restart-ключи и применив живой log.level (через ApplyWith, без записи и без рассылки). Читатели горячего пути — одна атомарная загрузка, без локов.
  • Сеть вне UI-потока. Клиент общается с UI только через потокобезопасный канал; лаг сети не морозит интерфейс.
  • Единственный писатель / читатель на сокет. На сервере — writeLoop; в клиенте — сериализация через clientMu. Рассылки неблокирующие: тормоз одного не задержит остальных.
  • Ограниченный рост. Reaper выселяет простаивающие вёдра rate-limit (TTL 10 мин); watcher снимает подписки на удалённые каталоги — два бывших источника неограниченного роста закрыты.
  • Единая точка авторизации. Таблица тип → минимальная роль в диспетчере — проверки прав не размазаны по обработчикам.
  • Один плохой пир не вредит другим. Malformed/oversize рвёт ровно одно соединение; таймаут рукопожатия закрывает полуоткрытые.
  • Проверено инструментом. Каждый пакет гоняется под go test -race; TSan нашёл и починил две гонки на M10, а «конкурентное применение конфига» — отдельный тест.

Тонкие места, помеченные в коде маркерами

маркерчто защищает
R3-2«поздний» DOWNLOAD_CANCEL отменяет закачку только если TransferID совпал с активной — иначе задел бы следующую.
R3-6флаг inFlight: сторож простоя не обрывает соединение, пока выполняется обработчик медленного запроса.
CR-06reserveUserSlot атомарно проверяет лимит сессий и метит сессию — параллельные входы одного логина не проскочат все.
CR-05дорукопожатные кадры ограничены малым потолком — аноним не заставит выделять мегабайты.
CR-10листинг, который превысил бы потолок кадра, заворачивается как ERROR(INTERNAL), а не шлётся oversize-кадром.
CR-09 · RR-5ключ checksum-кэша включает mtime в наносекундах и ctime — подмена файла тем же размером с восстановленным mtime не даст устаревшую сумму.
R4-3ChecksumCtx прерывает хеширование большого файла при отмене — закачку можно отменить, не досчитывая сумму минуту.
RR-1в рабочем цикле нет пофреймового дедлайна — таймаут никогда не «приземлится» посреди кадра и не рассинхронит поток.

Безопасность: честная модель угроз

Что защищает v2.0, а что — нет. Challenge–response защищает учётные данные: пароль и его хеш не пересекают сеть, кража users.json не даёт войти. Но содержимое трафика ещё не зашифровано — прослушивание байтов файла и MITM закрываются только TLS на этапе M14. Формат кадра уже готов к этому: TLS просто оборачивает TCP, кадры не меняются.

Что уже сделано

  • SCRAM-подобный вход (раздел 5): одноразовый 16-байтный challenge, доказательство HMAC-SHA256, сервер хранит только StoredKey, сравнение константного времени.
  • Растяжение пароля — PBKDF2-HMAC-SHA256, порог 600k итераций; тип хеша помечен в строке для прозрачной миграции.
  • Гард от перебора — временный бан IP после N неудач + пауза 1 с на каждый отказ; входы и баны аудируются.
  • Заключение путей в одной точке через os.Root — «..» и симлинки наружу не выведут за share-root (TOCTOU закрыт по построению на Linux-демоне).
  • Жёсткий админ-канал: ADMIN_SET берёт только белый список известных hot-ключей (никогда произвольный JSON), валидирует до применения (при ошибке ничего не меняется) и запрещает менять restart-ключи (порт, share_root) по сети — захваченная админ-сессия не перенаправит сервер в другой каталог. Само-kick запрещён.
  • Авторизация — на сервере: диспетчер отвергает ADMIN_* от не-админа как ACCESS_DENIED независимо от того, что нарисовал UI. Ограничения интерфейса — удобство, не защита.
  • Профили не хранят пароль вовсе. profiles.json (файл 0600) держит только метаданные подключения (хост/порт/логин/last-seen/каталог загрузок); поле Profile.Secret зарезервировано и всегда обнуляется перед записью — утечка файла профилей не раскроет пароль.

Сознательные ограничения v2.0

  • Нет upload (клиент → сервер) — появится с квотами на M13.
  • Нет TLS — трафик открытый; это осознанно отложено на M14.
  • Один сервер = один share-root; личные каталоги пользователей — M12+.
  • users.json — плоский JSON, пока пользователей единицы; SQLite приходит с квотами.
  • «Запомнить пароль» (хранение производного ключа + opt-in галка) — ещё не реализовано: поле Secret зарезервировано под эту будущую функцию, сейчас пароль на диск не попадает никогда.
  • Отложено (найдено ревью): медленный клиент может притормозить доставку событий остальным (правильный фикс — очередь-писатель на сессию, отложен до роста числа клиентов); watcher не переносит подписки при переименовании каталога — события под переименованным поддеревом кратко несут старый путь. Записано, чтобы не забыть.

Роадмап и «написать с нуля»

Этапы — вертикальные срезы: каждый заканчивается работающей системой. Порядок такой, чтобы яркий TUI появился как можно раньше, но не раньше, чем будет сервер, с которым он говорит. M7–M11 реализованы; M12–M14 — перспектива, под которую оставлены заделы в данных.

Чем Go-порт отличается от исходного ТЗ

  • Заключение путей — os.Root (Go 1.24+) вместо openat2(RESOLVE_BENEATH): заключение делегировано рантайму, а не написано руками.
  • Транспорт — горутина-на-соединение вместо epoll-реактора + пул потоков; лимит скорости тормозит через сон отправителя, а не backpressure EPOLLOUT (ядро оставлено готовым к epoll-порту).
  • Хеш пароля — PBKDF2-HMAC-SHA256 (порог 600k) вместо Argon2id/libsodium: стандартная сборка работает «из коробки».
  • TUI — Bubble Tea вместо FTXUI; наблюдение — fsnotify вместо inotify.

Дорожная карта «собрать такое самому»

  1. Кадр и кодек. 5-байтовый заголовок, big-endian, строгий require_end, проверка границ на каждом чтении. Юнит-тесты round-trip + отказ на битом вводе.
  2. VFS над одним корнем. os.Root, три формы пути, ленивый checksum-кэш по (path,size,mtime,ctime).
  3. Сессия и диспетчер. Машина состояний рукопожатия, таблица MinRole, единственный writeLoop, сторож простоя.
  4. SCRAM. Цепочка password → SaltedPassword → ClientKey → StoredKey, Proof/Verify константного времени.
  5. Передача. Отдельная горутина-стример, докачка по offset, checksum в конце, отмена по TransferID.
  6. Горячий конфиг. Плоские Settings, atomic.Pointer-снимок, валидация перед подменой, hot/restart-ключи.
  7. События. fsnotify + рекурсия + debounce → broadcast подписчикам неблокирующе.
  8. Клиент и TUI. Блокирующий транспорт с прозрачной пересылкой EVENT_*, Bubble Tea (Model–Update–View), сеть вне UI-потока, авто-реконнект.

Параллельно стоит читать: пакеты encoding/binary, os (os.Root), crypto/pbkdf2/hmac/subtle, golang.org/x/time/rate, fsnotify, фреймворк bubbletea; и подробную спецификацию порта в docs/tz/09-go-port.md.
