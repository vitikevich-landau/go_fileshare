# 01. Общая архитектура

## 1. Компоненты системы

```
┌─────────────────────────────┐          ┌──────────────────────────────────────────┐
│  fileshare-commander (TUI)   │          │  fileshare-daemon (Linux, Docker/systemd) │
│  ┌───────────┬───────────┐  │   TCP    │  ┌────────────────────────────────────┐  │
│  │ Локальная  │ Удалённая  │  │  proto  │  │ SessionManager (auth, роли, токены) │  │
│  │ панель (FS)│ панель     │◄─┼─────────┼─►│ VFS (дерево share-root, кэш метадан)│  │
│  └───────────┴───────────┘  │   v2     │  │ ServerCore v2 (реестр, статистика)   │  │
│  Cmdline · F-bar · диалоги   │          │  │ SettingsHub (горячий конфиг)         │  │
└─────────────────────────────┘          │  │ EventBus (push-уведомления клиентам) │  │
┌─────────────────────────────┐          │  │ RateLimiter (token bucket)           │  │
│  тот же бинарь, роль=admin   │◄────────┼─►│ AdminService (админ-канал)           │  │
│  + админ-панели              │          │  └────────────────────────────────────┘  │
└─────────────────────────────┘          │   epoll reactor + thread pool / coro      │
                                          │   config.json · users.json · share/       │
                                          └──────────────────────────────────────────┘
```

Три артефакта сборки:

| Артефакт | Было (v1) | Станет (v2) |
|---|---|---|
| `fileshare_server` | интерактивный процесс с admin-консолью в stdin | **демон** `fileshare-daemon`: без интерактивного stdin, логирование в stdout/journald, управление только по сети (админ-протокол) и сигналами |
| `fileshare_client` | REPL `connect/list/download` | **`fileshare-commander`** — полноэкранный TUI; старый REPL остаётся как `--batch`-режим для скриптов |
| `fileshare_loadtest` | нагрузочный инструмент | сохраняется, обновляется под протокол v2 |

## 2. Что переиспользуем из v1 (и что меняем)

| Модуль v1 | Судьба в v2 |
|---|---|
| `protocol.hpp/cpp` — фрейминг, `ByteReader`, big-endian сериализация | **Ядро сохраняется полностью.** Добавляются новые типы сообщений (см. [02-protocol-v2.md](02-protocol-v2.md)); контейнер `MessageType` расширяется, `MAX_CONTROL_PAYLOAD`/лимиты пересматриваются под списки директорий |
| `EpollServer` + `ThreadPool` (+ `CoroServer`) | **Сохраняются** как транспорт. Меняется только слой обработки сообщений (диспетчер per-session вместо per-connection stateless) |
| `ServerCore` | **Расширяется**: вместо `Catalog` — `Vfs`; добавляются `SessionManager`, `SettingsHub`, `EventBus`. Очередь stdin-команд удаляется (заменяется админ-протоколом) |
| `Catalog`/`config.json` (плоский список алиасов) | **Заменяется** на VFS над корневой директорией: раздаётся всё дерево `share_root`, ничего не надо `add`-ить вручную |
| `checksum` (CRC32/SHA-256) | Сохраняется. Checksum считается **лениво с кэшем** (по (path, size, mtime)), т.к. обходить всё дерево при старте дорого |
| `client.cpp` (логика скачивания, докачка `.part`) | Сохраняется как транспортный слой TUI-клиента |
| `net`-шим (Winsock/POSIX) | Сохраняется без изменений |
| admin-консоль в stdin | **Удаляется** — демону stdin недоступен; вся админка уезжает в протокол |

## 3. Слои сервера v2

```
             ┌──────────────────────────────────────────────┐
 транспорт   │  EpollServer / CoroServer (v1, без изменений)  │
             └──────────────────┬───────────────────────────┘
             ┌──────────────────▼───────────────────────────┐
 сессии      │  Session: состояние соединения ПОСЛЕ handshake │
             │  (кто вошёл, роль, подписки, текущая закачка)  │
             └──────────────────┬───────────────────────────┘
             ┌──────────────────▼───────────────────────────┐
 диспетчер   │  MessageDispatcher: Frame + Session → Handler  │
             │  проверка прав: user-сообщения / admin-сообщения│
             └───────┬──────────┬──────────┬────────────────┘
             ┌───────▼───┐ ┌────▼─────┐ ┌──▼──────────┐
 сервисы     │   Vfs      │ │ Transfer  │ │ AdminService │
             │ (дерево,   │ │ (стриминг,│ │ (get/set     │
             │  метаданные│ │  offset,  │ │  конфига,    │
             │  checksum- │ │  rate     │ │  clients,    │
             │  кэш)      │ │  limiter) │ │  kick, stats)│
             └───────┬───┘ └────┬─────┘ └──┬──────────┘
             ┌───────▼──────────▼───────────▼─────────────┐
 состояние   │ SettingsHub (снапшоты) · SessionManager ·    │
             │ EventBus · ClientRegistry · Stats            │
             └─────────────────────────────────────────────┘
```

Принципы:

- **Session** — центральная сущность v2. До успешного `AUTH` соединение находится в
  состоянии `HANDSHAKE` и принимает только `HELLO`/`AUTH_*`/`PING`; всё остальное —
  `ERROR(AUTH_REQUIRED)` + разрыв. Таймаут на handshake (по умолчанию 10 с) защищает
  от висящих полуоткрытых соединений.
- **Диспетчер** — единая точка авторизации: у каждого типа сообщения объявлен
  минимальный уровень (`anonymous` / `user` / `admin`). Не надо размазывать проверки
  по обработчикам.
- **Сервисы не знают про транспорт**: как и в v1, «что делает сервер» отделено от
  «как двигаются байты» — это уже сработало для трёх движков, сохраняем.

## 4. Слои клиента

```
┌─────────────────────────────────────────────┐
│ UI (FTXUI): экраны, панели, диалоги, клавиши  │   ← только отрисовка и ввод
├─────────────────────────────────────────────┤
│ AppState: панели, курсоры, выделение,        │   ← вся логика UI-состояния,
│ очередь передач, лог ошибок, тема            │      тестируется без терминала
├─────────────────────────────────────────────┤
│ RemoteFs / LocalFs — общий интерфейс IFs      │   ← панель не знает, локальная она
│ (list, stat, download, watch-события)         │      или удалённая
├─────────────────────────────────────────────┤
│ Connection: сокет, протокол v2, фоновый поток │   ← сеть строго вне UI-потока
│ приёма, реконнект, очередь исходящих          │
└─────────────────────────────────────────────┘
```

Ключевое правило: **сетевой ввод-вывод никогда не выполняется в UI-потоке.**
`Connection` живёт в своём потоке, общение с UI — через потокобезопасную очередь
событий (`std::deque` + mutex + пробуждение event loop FTXUI через
`ScreenInteractive::PostEvent`). Иначе любой лаг сети заморозит интерфейс.

## 5. Потоки данных: типовые сценарии

### Вход и первый экран
1. TUI показывает форму подключения (хост/порт/логин/пароль, «запомнить»).
2. `Connection` устанавливает TCP, шлёт `HELLO(proto=2, client="commander/2.0")`.
3. Сервер отвечает `HELLO_OK(server, proto=2, auth=challenge)` + соль/челлендж.
4. Клиент шлёт `AUTH_REQUEST(login, HMAC(challenge, password_hash))`.
5. Сервер: `AUTH_OK(role, session_token, server_time, motd)` — TUI переключается на
   двухпанельный экран; правая панель сразу шлёт `LIST_DIR("/")`.
6. Клиент подписывается на события: `SUBSCRIBE(events=fs,transfers)`.

### Скачивание (F5 с удалённой панели на локальную)
1. UI ставит задачу в `TransferQueue` (последовательное выполнение, очередь видна
   в отдельной панели).
2. `Connection` шлёт `DOWNLOAD_REQUEST(path, offset)` (offset > 0, если есть `.part`).
3. Поток приёма пишет `CHUNK_DATA` в `<имя>.part`, публикует прогресс в UI-очередь.
4. `DOWNLOAD_DONE(checksum)` → сверка → переименование `.part` → зелёная строка в
   логе операций; ошибка → красная строка + файл остаётся `.part` для докачки.

### Живое изменение настройки админом
1. Админ в TUI меняет «лимит скорости на клиента» → `ADMIN_SET("limits.per_client_bps", v)`.
2. `AdminService` валидирует, собирает новый `Settings`, атомарно подменяет снапшот
   в `SettingsHub`, пишет конфиг на диск, рассылает `EVENT(config_changed)` подписчикам.
3. Все активные передачи на следующем чанке читают новый снапшот — скорость
   изменилась **без разрыва соединений и без перезапуска**.

## 6. Технологический стек v2 (дельта к v1)

| Область | Выбор | Обоснование |
|---|---|---|
| TUI | **FTXUI** (FetchContent) | C++20, работает на Windows и Linux (наш клиент кроссплатформенный), декларативные layout'ы, MIT. Запасной план: ncurses + ручной layout (больше кода, зато везде есть) |
| Хэширование паролей | **libsodium** (Argon2id) | стандарт де-факто; fallback — PBKDF2 на уже имеющемся SHA-256, если не хочется новой зависимости на первом этапе |
| Конфиг | nlohmann/json (как в v1) | уже в проекте; формат описан в [03-server-daemon.md](03-server-daemon.md) |
| Наблюдение за файлами | inotify (Linux) | для событий «появился новый файл» в share-root и для отслеживания правки config.json руками |
| Логи | свой тонкий логгер поверх stdout (уровни, timestamp) | journald/docker собирают stdout; тащить spdlog не обязательно, но допустимо |
| БД пользователей | `users.json` на первом этапе → SQLite на этапе квот | пока пользователей единицы — JSON достаточно и прозрачен |

## 7. Осознанные ограничения v2.0

- Upload (клиент → сервер) — **не** в первой версии; появляется вместе с квотами (M13).
- TLS — отдельный этап после challenge–response (см. [06-security.md](06-security.md)).
- Никакого P2P, как и раньше.
- Один сервер = один share-root (личные каталоги пользователей — этап M12+).
