# 05. Админ-режим

## 1. Принцип

Отдельного «админ-клиента» как бинаря нет — это **тот же fileshare-commander**.
Различие определяется ролью, которую сервер вернул в `AUTH_OK`:

- `role=user` — обычные две панели, F9 показывает «недостаточно прав»;
- `role=admin` — всё то же + `F9` открывает админ-панель; в F-баре появляется
  `9Админ`, в заголовке — пометка `[admin]`.

Авторизация — **на сервере**: диспетчер сообщений отклоняет `ADMIN_*` от
не-админской сессии кодом `ACCESS_DENIED`, независимо от того, что нарисовал UI.
UI-ограничения — только удобство, не защита.

## 2. Админ-панель (F9)

Полноэкранный режим в теме «неоновый грид» (см. [04-tui-client.md](04-tui-client.md) §4).
Кадр фиксированной структуры: верхний бар (логотип · сервер · статус линка),
рейка вкладок с неоновым подчёркиванием активной, тело вкладки, строка-тост
с результатом последней операции и футер с клавишами. Вкладки переключают
Tab/цифры. Все диалоги (kick, shutdown, детали сессии, меню F2, редактор
настроек) — модальные окна ПО ЦЕНТРУ экрана поверх контента, а не строки,
приклеенные к низу.

```
 █▓▒░ FSHARE ADMIN  ◢ vps.example.com          ● LINK:ONLINE  ROLE:ADMIN
  1 OVERVIEW   2 CLIENTS   3 SETTINGS   4 JOURNAL
─━━━━━━━━━━━━───────────────────────────────────────────────────────────
```

### 2.1. Вкладка «Обзор» (`ADMIN_STATS`, автообновление раз в 2с)

Сетка карточек телеметрии с цветовым кодом (на узких экранах — компактный
список тех же значений):

```
 ╭──────────────────╮ ╭──────────────────╮ ╭──────────────────╮
 │ SESSIONS         │ │ TRANSFERS        │ │ TRAFFIC          │
 │ 12               │ │ 3                │ │ 811.4G           │
 │ active conns     │ │ 1204 completed   │ │ bytes sent total │
 ╰──────────────────╯ ╰──────────────────╯ ╰──────────────────╯
 …SHARE · UPTIME · CORE, ниже — LIMIT/PER-CLIENT и LIMIT/GLOBAL
```

### 2.2. Вкладка «Клиенты» (`ADMIN_LIST_CLIENTS`)

Таблица: id, логин, IP, роль, аптайм сессии, текущая операция
(`качает /video/big.iso, 43%, 12 MB/s`), передано байт.
Клавиши: `F8`/`k` — kick с подтверждением (`ADMIN_KICK`), `Enter` — детали сессии.

### 2.3. Вкладка «Настройки» — живая конфигурация

Список параметров из `ADMIN_GET_CONFIG` с бейджами hot/restart:

```
 KEY                        VALUE        MODE
 limits.per_client_bps      10 MB/s      ⚡ hot     ← Enter — изменить
 limits.global_bps          50 MB/s      ⚡ hot
 limits.max_connections     200          ⚡ hot
 limits.idle_timeout_s      600          ⚡ hot
 server.motd                "Добро…"     ⚡ hot
 server.port                5555         ⟳ restart  (серым: менять по сети нельзя)
 log.level                  info         ⚡ hot
```

- Enter → диалог редактирования с валидацией формата на клиенте
  («10M», «0 = без лимита»);
- подтверждение → `ADMIN_SET(key, value)` → результат зелёным/красным;
- параметры `restart` показываются, но редактирование заблокировано с пояснением —
  **порт и share_root по сети менять нельзя** (безопасность: иначе угнанная
  админ-сессия перенаправит сервер на чужой каталог);
- если конфиг поменял кто-то другой (второй админ, SIGHUP) — вкладка обновляется
  сама по `EVENT_CONFIG` (новое значение подставляется на месте), а сам факт
  изменения уходит во вкладку «Журнал». Мигающих подсветок нет: тема осознанно
  отказывается от мигания (см. [04-tui-client.md](04-tui-client.md) §4).

### 2.4. Вкладка «Журнал»

Живой хвост серверных событий уровня warn/error + аудит изменений конфига
(`EVENT_NOTICE` + `EVENT_CONFIG`): кто вошёл, кого кикнули, что поменяли.

### 2.5. Управление жизненным циклом

Через меню (F2) в админ-панели:
- **Graceful shutdown** (`ADMIN_SHUTDOWN(grace)`) — двойное подтверждение с вводом
  слова `shutdown`; всем клиентам предварительно рассылается
  `EVENT_NOTICE("сервер остановится через N с")`;
- **Reload users** — форс-перечитка users.json.

## 3. Серверная сторона: AdminService

```cpp
class AdminService {
public:
    AdminService(SettingsHub&, SessionManager&, Stats&, EventBus&);
    std::string   get_config_json() const;             // эффективный конфиг + пометки hot/restart
    ApplyResult   set(std::string key, std::string json_value, const Session& who);
    std::vector<SessionInfo> list_clients() const;
    bool          kick(std::uint64_t session_id, const Session& who);
    StatsSnapshot stats() const;
    void          shutdown(std::uint32_t grace_s, const Session& who);
};
```

Правила:
- **Белый список ключей.** `set()` принимает только известные hot-ключи; всё
  прочее — `ACCESS_DENIED` с пояснением. Никакого «положить произвольный JSON».
- **Валидация значений до применения**: диапазоны (`per_client_bps ≤ global_bps`,
  таймауты > 0), типы. Ошибка — атомарно ничего не меняется.
- **Аудит**: каждое действие логируется (`admin 'vit' set limits.per_client_bps: 10M → 20M from 1.2.3.4`)
  и уходит в `EVENT_CONFIG`/`EVENT_NOTICE`.
- Kick самого себя — запрещён (защита от «выпилил себя и потерял доступ»);
  kick другого админа — разрешён, но логируется отдельно.

## 4. Аварийный доступ (если сеть недоступна/сломали конфиг)

Демон без stdin, поэтому аварийные пути:
1. `SIGHUP` — перечитать исправленный руками config.json;
2. `fileshare-daemon --check-config` — провалидировать перед перезапуском;
3. `--add-user` / `--reset-password LOGIN` — локальная утилита на сервере
   (доступ по SSH к VPS считается доверенным периметром).
