# 06. Безопасность

## 1. Модель угроз (честная, по этапам)

| Угроза | v2.0 (challenge–response) | v2.x (TLS, M14) |
|---|---|---|
| Пассивное прослушивание пароля | ✅ пароль не передаётся | ✅ |
| Прослушивание содержимого файлов | ❌ трафик открыт | ✅ |
| MITM / подмена сервера | ❌ | ✅ (проверка сертификата) |
| Перебор пароля по сети | ✅ бан после N неудач + медленный KDF | ✅ |
| Кража users.json с сервера | ✅ Argon2id, пароль не восстановить | ✅ |
| Кража profiles.json с клиента | ⚠ вход на этот сервер возможен (см. §4) | ⚠ то же |

Вывод: v2.0 защищает **учётные данные**, но не **содержимое трафика**. Для личного
VPS с некритичными файлами это осознанный первый шаг; TLS — запланированный этап,
формат протокола к нему готов (TLS оборачивает TCP, фреймы не меняются).

## 2. Хранение паролей на сервере

- `users.json` хранит **Argon2id-хэш** (libsodium, `crypto_pwhash_str`):
  соль и параметры внутри строки, устойчиво к перебору на GPU.
- Fallback без libsodium (флаг сборки `FILESHARE_USE_SODIUM=OFF`):
  PBKDF2-HMAC-SHA-256 (итерации ≥ 600k) на уже имеющемся в проекте SHA-256 —
  чтобы первый этап не блокировался новой зависимостью. В конфиге users.json
  тип хэша указан в самой строке (`$argon2id$` / `$pbkdf2$`), миграция —
  прозрачная перезапись при следующем входе.

## 3. Challenge–response (почему не «пароль по сети»)

Схема (вариант SCRAM-подобной, упрощённый):

```
1. Клиент → HELLO
2. Сервер → HELLO_OK(challenge = 16 случайных байт, nonce одноразовый, TTL 30с)
3. Клиент: key   = KDF(password)              // та же KDF, что на сервере — параметры
                                               // сервер сообщает в HELLO_OK
          proof = HMAC-SHA-256(key, challenge || login)
   Клиент → AUTH_REQUEST(login, proof)
4. Сервер: сверяет HMAC с тем же key из users.json → AUTH_OK / AUTH_FAIL
```

Свойства:
- пароль и его хэш по сети не ходят; challenge одноразовый → перехваченный proof
  бесполезен (replay невозможен);
- сервер обязан хранить `key` (производную), а не только Argon2-строку → в
  users.json хранится и verifier для HMAC (как в SCRAM: `StoredKey`);
- защита от перебора: после `limits.auth_fail_ban_s` неудач — временный бан IP,
  задержка 1с на каждый AUTH_FAIL.

Примечание: полноценный SCRAM-SHA-256 (RFC 5802) — хороший ориентир; если
захочется строгости, реализуем его дословно, структура сообщений уже совпадает.

## 4. Хранение пароля на клиенте («запомнить пароль»)

- Хранится `key` (производная KDF), не пароль: утечка не раскрывает пароль,
  который пользователь мог использовать где-то ещё.
- Но `key` достаточно для входа на **этот** сервер → файл `0600`, галка по
  умолчанию выключена, в UI — честное предупреждение.
- Улучшение вне scope v2.0: системные хранилища (libsecret / Windows DPAPI).

## 5. Прочие меры на сервере

- **Валидация путей** в единственной точке (`Vfs::resolve`): нормализация, запрет
  `..`, realpath-контроль симлинков — защита от directory traversal.
- **Лимиты протокола** (наследие v1): oversize/битые фреймы рвут только одно
  соединение; handshake-таймаут против slow-loris; `max_connections`,
  `max_sessions_per_user`.
- **Привилегии процесса**: демон работает под отдельным пользователем `fileshare`,
  share-root ему read-only (до появления upload); systemd-hardening / контейнер.
- **Аудит**: все входы (успех/неудача, IP), все админ-действия — в лог.

## 6. TLS (этап M14, план)

- OpenSSL уже опциональная зависимость проекта (SHA-256) → берём её же.
- Обёртка на уровне транспорта: `TlsConnection` поверх сокета, фреймы протокола
  не меняются. Для epoll — интеграция неблокирующего handshake в машину состояний
  (`SSL_ERROR_WANT_READ/WRITE` → те же EPOLLIN/EPOLLOUT переходы).
- Сертификат: самоподписанный + **pinning в профиле клиента** (TOFU: при первом
  подключении показать fingerprint, сохранить, при смене — громкое красное
  предупреждение). Let's Encrypt — если у VPS будет доменное имя.
- После TLS challenge–response остаётся (защита users.json + взаимная независимость
  слоёв).
