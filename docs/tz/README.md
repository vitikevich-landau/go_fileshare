# fileshare v2 — «FShare Commander»: ТЗ и план реализации

> 📖 Быстрый визуальный обзор всей системы — интерактивная страница
> [`../interactive/index.html`](../interactive/index.html): карта пакетов, кадр
> протокола по байтам, карта сообщений, пошаговый SCRAM и сквозные сценарии.
> Документы ниже — это исходное ТЗ и подробный план реализации.


Эволюция текущего проекта (C++20 файлообменник со своим бинарным протоколом) в
полноценную клиент-серверную систему:

- **Сервер** — постоянно работающий процесс (демон/служба) на Linux: локально в Docker,
  затем на VPS (Timeweb Cloud). Раздаёт по своему протоколу **дерево директорий**,
  а не плоский список файлов. Управляется на лету, **без перезапуска**.
- **Клиент** — полноэкранный консольный **TUI в стиле Midnight Commander**: две панели,
  командная строка, горячие клавиши, полоса функциональных клавиш, экран логина,
  индикация подключения/загрузки, цветовая подсветка нового и ошибок.
- **Админ-клиент** — тот же командер, но после входа под админской учёткой открывает
  админ-панель: конфигурация сервера на лету (скорость, лимиты), список клиентов,
  kick, статистика, позже — управление пользователями.
- **Перспектива** — многопользовательский режим: учётные записи, роли, квоты,
  изоляция пользователей (личные каталоги в духе «Яндекс.Диска»).

## Ключевые решения (кратко)

1. **Протокол v2 несовместим с v1, но построен на том же фрейминге** (5-байтовый
   заголовок + payload, big-endian). Первым сообщением идёт `HELLO` с версией —
   сервер v2 корректно отказывает клиенту v1 и наоборот.
2. **Ядро сервера (`ServerCore`) сохраняется и расширяется**, транспорт остаётся
   epoll + пул потоков (и опционально корутины). Каталог заменяется на
   **VFS-слой** над корневой директорией раздачи.
3. **Горячая конфигурация** — через снапшоты настроек (`std::shared_ptr<const Settings>`,
   атомарная подмена): рабочие потоки читают актуальный снапшот на каждой итерации,
   поэтому изменение (например, лимита скорости) применяется мгновенно и без локов
   на горячем пути. Источники изменений: админ-протокол, `SIGHUP`, наблюдение за
   файлом конфига.
4. **TUI на [FTXUI](https://github.com/ArthurSonzogni/FTXUI)** (C++20, header-only-дух,
   кроссплатформенно Windows/Linux, активно развивается). Запасной вариант — ncurses.
5. **Аутентификация challenge–response** (пароль не ходит по сети открытым текстом),
   хэши паролей на сервере — Argon2id (libsodium). TLS — отдельным этапом.
6. **Один бинарь клиента** для пользователя и админа: админ-функции открываются по
   роли, полученной при входе.
