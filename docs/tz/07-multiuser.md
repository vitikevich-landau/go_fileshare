# 07. Перспектива: пользователи, роли, квоты («мини-Яндекс.Диск»)

Этот документ — **проектный задел** (этапы M12–M13), не обязательства v2.0.
Цель: зафиксировать целевую модель сейчас, чтобы решения v2.0 ей не противоречили.

## 1. Модель данных

```
User:   id · login · pass (Argon2id/verifier) · role · enabled · created_at
Role:   admin | user           (v2.0)
        + uploader | readonly  (M12: гранулярность)
Space:  user_id · quota_bytes · used_bytes
Grant:  user_id · vpath · rights (r / rw)     (M12: шаринг между пользователями)
```

Хранилище: `users.json` → **SQLite** на этапе M12 (транзакционность нужна квотам:
`used_bytes` должен обновляться атомарно с окончанием записи файла).

## 2. Пространства и изоляция

Плоская модель «один share-root для всех» превращается в:

```
/srv/fileshare/
├── public/            # видно всем (то, что раздаётся сейчас)
├── homes/
│   ├── vit/           # личное пространство: квота, полный доступ владельца
│   └── anna/
└── shared/            # M12+: каталоги, расшаренные грантами
```

Виртуальный корень зависит от пользователя — VFS монтирует:

```
/ ─┬─ public   → /srv/fileshare/public      (r для всех)
   ├─ home     → /srv/fileshare/homes/<login>  (rw владельцу, невидим другим)
   └─ shared/… → по таблице Grant
```

**Изоляция обеспечивается на сервере** (VFS не отдаст чужой `home` ни при каком
пути — проверка на этапе `resolve`), а не «скрытием» в UI. Это прямой ответ на
требование «чтобы пользователи между собой не перепрыгивали».

## 3. Квоты

- Квота — на `home` пользователя (байты). Проверка **до** начала upload:
  `UPLOAD_REQUEST(size)` → сервер сверяет `used + size ≤ quota` →
  `QUOTA_EXCEEDED` сразу, а не после передачи гигабайта.
- `used_bytes` пересчитывается фоновым обходом при старте (self-healing) и
  инкрементально поддерживается на каждой записи/удалении.
- В клиенте: полоска квоты в статусе панели `home` (`▓▓▓░░ 6.2G / 10G`),
  жёлтая при > 80%, красная при > 95%.
- Rate-лимиты тоже становятся per-role/per-user (таблица в конфиге:
  `limits.roles.user.per_client_bps`).

## 4. Upload (без него квоты бессмысленны)

Зеркально download'у, протокол зарезервирован в группе `0x3x`:

```
UPLOAD_REQUEST (path, size, mtime)  → UPLOAD_ACCEPT (transfer_id, offset)
                                       // offset > 0 = докачка недозалитого
UPLOAD_CHUNK   (transfer_id, bytes) → …
UPLOAD_DONE    (transfer_id, checksum) → UPLOAD_RESULT (ok / checksum mismatch)
```

- Сервер пишет во временный файл (`.upart`), переименовывает после сверки
  checksum — на диске никогда не бывает «наполовину залитых» файлов под финальным
  именем.
- В TUI: F5 с локальной панели на удалённую `home` — полная симметрия с download,
  F6 — переместить, F7 — mkdir, F8 — удалить (в пределах прав).

## 5. Администрирование пользователей (M12, вкладка в админ-панели)

`ADMIN_USER_*` сообщения (`0x5A–0x5F`): list / add / disable / set-role /
set-quota / reset-password. Всё через тот же AdminService с аудитом.

## 6. Что для этого уже заложено в v2.0 (чтобы потом не переделывать)

- `role:u8` в `AUTH_OK` — enum расширяется без слома формата.
- `transfer_id` в передачах — параллельные и восстановимые передачи.
- Код ошибки `QUOTA_EXCEEDED` зарезервирован.
- VFS — единственная точка резолва путей: подмена «один корень» → «виртуальные
  маунты по пользователю» локализована в одном классе.
- `SUBSCRIBE`/`EVENT_FS` уже фильтруются по видимости сессии (в v2.0 видимость у
  всех одинаковая, но фильтр — в сигнатуре EventBus с первого дня).
